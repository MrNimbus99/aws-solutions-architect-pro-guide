Quiz 1 — Sample Question
Domain 1: Design Solutions for Organizational Complexity
Select THREE. A large enterprise is designing a network architecture for a new AWS deployment with the following requirements: 100+ VPCs across 4 Regions, centralized egress to the internet, centralized firewall inspection, hybrid connectivity to on-premises data centers, and support for future growth to 500+ VPCs. Which THREE components are essential for this architecture?
  • ✓ AWS Direct Connect with Direct Connect Gateway for hybrid connectivity
  • VPC peering connections between all VPCs
  • NAT Gateways in every VPC for internet egress
  • ✓ AWS Transit Gateway in each Region with inter-Region peering
  • ✓ AWS Network Firewall in centralized inspection VPCs
  • AWS PrivateLink for all inter-VPC communication
Show Explanation

- Why correct (A, D, E): Transit Gateway provides scalable VPC connectivity and centralized routing management, supporting hundreds of VPCs with inter-Region peering. Direct Connect with Direct Connect Gateway provides high-bandwidth, low-latency hybrid connectivity across multiple Regions. Network Firewall in centralized inspection VPCs provides managed firewall capabilities for traffic inspection. These three components form the foundation of a scalable, secure enterprise network architecture. - Why B is wrong: VPC peering does not scale to 100+ VPCs (requires thousands of peering connections for full mesh). Peering does not support transitive routing or centralized management. Transit Gateway is the appropriate solution for large-scale VPC connectivity. - Why C is wrong: NAT Gateways in every VPC increase costs and do not support centralized egress architecture. The requirement specifies centralized egress, which requires routing internet-bound traffic through centralized egress VPCs via Transit Gateway. - Why F is wrong: PrivateLink is designed for exposing specific services to other VPCs or accounts, not for general inter-VPC communication. It operates on a service-consumer model and does not provide full network-level connectivity between VPCs. Transit Gateway is the appropriate solution for VPC-to-VPC connectivity at scale.

Domain 1: Design Solutions for Organizational Complexity
A healthcare organization must maintain audit logs of all API activity across 60 AWS accounts for 7 years to meet regulatory compliance requirements. Logs must be tamper-proof, searchable, and stored cost-effectively. The security team requires alerts when specific high-risk API calls are made (e.g., security group changes, IAM policy modifications). Which architecture meets these requirements?
  • Enable CloudTrail in each account with logs stored in individual S3 buckets, enable S3 Object Lock for tamper-proofing, and use Athena for searching
  • Enable AWS Config in all accounts to record API activity and use Config rules to detect high-risk changes
  • Enable CloudTrail in each account with logs sent to CloudWatch Logs, use CloudWatch Logs Insights for searching, and create CloudWatch alarms for high-risk API calls
  • ✓ Enable CloudTrail organization trail in the management account with logs stored in a centralized S3 bucket with S3 Object Lock, use S3 Intelligent-Tiering for cost optimization, and configure EventBridge rules for high-risk API alerts
Show Explanation

- Why D is correct: CloudTrail organization trail provides centralized logging across all accounts from a single configuration. S3 Object Lock in compliance mode ensures tamper-proof storage that cannot be overwritten or deleted, even by the root user. S3 Intelligent-Tiering automatically moves logs to cost-effective storage tiers, which is ideal for 7-year retention. Athena enables ad-hoc SQL querying of logs stored in S3. EventBridge rules can match specific API call patterns (e.g., AuthorizeSecurityGroupIngress, PutUserPolicy) and trigger SNS alerts. This architecture meets all requirements — centralization, tamper-proofing, searchability, cost-efficiency, and alerting — with minimal operational overhead. - Why A is wrong: Individual CloudTrail trails in each account create management overhead across 60 accounts. Managing 60 separate S3 buckets with individual Object Lock configurations is operationally inefficient. There is no centralized visibility without additional aggregation. Organization trails eliminate this complexity by automatically applying to all accounts from a single trail. - Why B is wrong: AWS Config records resource configuration changes (e.g., a security group rule was modified), not all API activity. Config does not capture data-plane API calls or management-plane calls like DescribeInstances. CloudTrail is the correct service for comprehensive API activity logging. Config is designed for configuration compliance auditing, not API audit logging. - Why C is wrong: CloudWatch Logs is significantly more expensive than S3 for long-term storage, especially over a 7-year retention period. CloudWatch Logs pricing is based on ingestion and storage, which would be cost-prohibitive at this scale. S3 with Intelligent-Tiering provides much more cost-effective long-term storage for compliance logs.

Select TWO. An e-commerce company operates a shopping cart application that must handle Black Friday traffic spikes of 50x normal load. The application uses EC2 instances, RDS MySQL, and ElastiCache. Last year, the database became a bottleneck during peak traffic. Which TWO actions improve database scalability for traffic spikes?
  • Enable RDS Multi-AZ for better performance
  • Disable database backups during peak traffic
  • ✓ Implement RDS read replicas and configure the application to route read traffic to replicas
  • Migrate to DynamoDB for all data storage
  • Increase the RDS instance size to the largest available
  • ✓ Implement database query result caching in ElastiCache to reduce database load
Show Explanation

- Why correct: RDS read replicas (C) distribute read traffic across multiple database instances, reducing load on the primary database. ElastiCache caching (F) reduces database queries by serving frequently accessed data from memory. Together, these approaches significantly reduce database load during traffic spikes, addressing the bottleneck. Both are proven patterns for scaling read-heavy workloads. - Why A is wrong: RDS Multi-AZ provides high availability through synchronous standby replication and automatic failover, but the standby instance does not serve read traffic. Multi-AZ improves availability, not read performance or scalability. It does not address the database bottleneck during traffic spikes. - Why B is wrong: Disabling backups during peak traffic is a dangerous practice that violates data protection requirements. Automated backups have minimal performance impact on RDS MySQL, and disabling them creates unacceptable risk of data loss without meaningful performance gains. - Why D is wrong: Migrating to DynamoDB is a major architectural change that requires significant application refactoring of SQL queries, schema design, and data access patterns. While DynamoDB scales well, the migration effort and risk make this impractical for addressing an upcoming traffic spike. - Why E is wrong: Increasing the RDS instance size (scale-up) provides more capacity but has hard upper limits and is less flexible than scale-out with read replicas. A single larger instance cannot handle 50x traffic increases, and scale-up alone does not distribute the read workload. Combining scale-out (read replicas) with caching is more effective and cost-efficient.

Domain 1: Design Solutions for Organizational Complexity
A large enterprise with three business units is designing a multi-account AWS environment. Each business unit has development, staging, and production workloads. The security team requires that all production accounts across every business unit have identical restrictive SCPs applied, while development accounts need more permissive policies to allow experimentation. Which organizational unit (OU) hierarchy meets these requirements with the LEAST administrative overhead?
  • Create one OU per business unit, with child OUs for each environment underneath, and apply production SCPs to each business unit's production child OU separately
  • ✓ Create top-level OUs by environment (Production, Staging, Development) and place each business unit's accounts into the corresponding environment OU, applying SCPs at the environment OU level
  • Place all accounts in the root OU and use tag-based IAM policies to differentiate between production and development permissions
  • Create a separate AWS Organization for each environment and use cross-organization IAM role assumption for shared access
Show Explanation

- Why correct: An environment-first OU hierarchy (Production OU, Staging OU, Development OU at the top level) allows applying a single set of restrictive SCPs to the Production OU, which automatically applies to all production accounts regardless of business unit. Development accounts receive more permissive SCPs at the Development OU level. This eliminates maintaining duplicate SCPs across multiple OUs and reduces administrative overhead as additional business units are onboarded. - Why A is wrong: A business-unit-first hierarchy requires applying identical production SCPs to each business unit's production child OU separately. With three business units, you manage three copies of the same production SCPs. As business units grow, administrative overhead increases linearly because each new business unit needs its own production child OU with replicated SCPs. - Why C is wrong: Placing all accounts in the root OU prevents applying different SCPs to different environments, since SCPs at the root apply to all accounts equally. Tag-based IAM policies can be modified by account administrators and do not provide the centralized enforcement that SCPs applied to OUs deliver. - Why D is wrong: Separate AWS Organizations prevent consolidated billing, centralized governance, and unified CloudTrail logging. Cross-organization role assumption adds significant complexity and eliminates the benefits of a single Organization such as SCP inheritance, consolidated Cost Explorer, and organization-wide service integrations.

Quiz 6 — Sample Question
Domain 1: Design Solutions for Organizational Complexity
A multinational company operates 12 VPCs across two AWS Regions. Each VPC belongs to a different business unit. The company requires any VPC to communicate with any other VPC, including cross-Region, with centralized route management and minimal operational overhead. Which networking architecture meets these requirements?
  • Create individual VPC peering connections between all VPCs and use route tables to direct traffic
  • Deploy a shared VPC and move all workloads into a single VPC with separate subnets per team
  • ✓ Deploy AWS Transit Gateway in the hub Region with inter-Region peering to a second Transit Gateway, and attach each VPC to its regional Transit Gateway
  • Establish AWS Site-to-Site VPN tunnels between every pair of VPCs across Regions
Show Explanation

- Why correct: Transit Gateway acts as a regional network hub that supports transitive routing between all attached VPCs. Inter-Region peering connects two Transit Gateways across Regions, enabling cross-Region VPC-to-VPC communication without managing individual peering connections. This scales well and simplifies route management as VPCs are added. - Why A is wrong: VPC peering does not support transitive routing. With 12 VPCs, you would need up to 66 peering connections (n*(n-1)/2), and cross-Region peering still wouldn't allow traffic to transit through an intermediary VPC. This creates an unmanageable mesh that does not scale. - Why B is wrong: A shared VPC centralizes networking but forces all workloads into a single VPC, which violates blast-radius isolation between business units and makes independent scaling and security segmentation difficult. It also cannot span multiple Regions within a single VPC. - Why D is wrong: Site-to-Site VPN connections between VPCs would require deploying VPN endpoints in every VPC, creating a full mesh of encrypted tunnels. This adds latency, is operationally complex, and VPN throughput (up to 1.25 Gbps per tunnel) is much lower than Transit Gateway throughput. VPN is designed for on-premises-to-AWS connectivity, not VPC-to-VPC.

Quiz 7 — Sample Question
Domain 2: Design for New Solutions
A company runs a business-critical three-tier web application on AWS. The application consists of an Application Load Balancer, an Auto Scaling group of EC2 instances, and an Amazon Aurora database. The operations team needs to deploy application updates with zero downtime and the ability to roll back within minutes if issues are detected. The team manages all infrastructure using AWS CloudFormation. Which deployment approach meets these requirements?
  • Update the CloudFormation stack in-place and manually terminate old EC2 instances after verifying the new deployment
  • ✓ Create a new CloudFormation stack with the updated resources, use Amazon Route 53 weighted routing to gradually shift traffic to the new stack, and delete the old stack after validation
  • Deploy updates directly to the existing EC2 instances using SSH and shell scripts
  • Delete the existing CloudFormation stack and create a new one with the updated template
Show Explanation

- Why correct: Creating a parallel CloudFormation stack implements a blue/green deployment pattern. Route 53 weighted routing enables gradual traffic shifting (e.g., 10% → 50% → 100%) to the new environment while keeping the old stack intact as a rollback target. If issues are detected during validation, traffic can be shifted back to the original stack within minutes by updating the Route 53 weights. This provides true zero-downtime deployment with instant rollback capability. - Why A is wrong: In-place CloudFormation stack updates trigger rolling replacements of resources, which can cause brief service disruptions during the update process. Manual instance termination is error-prone and slow. If the update introduces a bug, rollback requires another full stack update, which takes time and is not 'within minutes.' This approach does not guarantee zero downtime. - Why C is wrong: Deploying updates directly via SSH completely bypasses Infrastructure as Code principles. The CloudFormation template and actual infrastructure become out of sync (configuration drift), making future deployments unpredictable. There is no automated rollback mechanism — any issues require manual intervention on each instance. This approach is not repeatable, auditable, or scalable. - Why D is wrong: Deleting the existing stack before creating a new one causes complete application downtime during the transition period. There is no rollback path because the original environment no longer exists. This directly violates the zero-downtime requirement and introduces significant risk for a business-critical application.

Which DR strategy meets these requirements?
  • Deploy a full production-scale environment in us-west-2 with active-active traffic distribution using Route 53 weighted routing
  • Configure nightly database backups to an S3 bucket in us-west-2 with cross-region replication, and maintain CloudFormation templates to rebuild the infrastructure during a disaster
  • ✓ Maintain a pilot light environment in us-west-2 with continuously replicated databases using Aurora Global Database and pre-configured but stopped EC2 instances that can be started and scaled during failover
  • Deploy identical Auto Scaling groups in us-west-2 running at 50% of production capacity with Aurora Global Database replication and Route 53 failover routing
Show Explanation

- Why correct: The pilot light strategy keeps critical data services (Aurora Global Database) continuously replicated while minimizing compute costs by keeping EC2 instances stopped. Quiz 9 — Sample Question
Domain 2: Design for New Solutions
A company is designing a multi-tier web application on AWS. The application tier runs on EC2 instances in private subnets and must communicate with an Amazon RDS MySQL database in a separate private subnet. The security team requires that only the application tier can reach the database on port 3306, and no other traffic should be allowed to the database subnet. The solutions architect also needs to block specific IP ranges known for malicious activity from reaching the public-facing ALB. Which combination of network controls meets these requirements?
  • Configure the RDS security group to allow inbound on port 3306 from the application tier security group. Add a network ACL on the database subnet to deny inbound traffic from the known malicious IP ranges.
  • ✓ Configure the RDS security group to allow inbound on port 3306 from the application tier security group. Add a network ACL on the public subnet to deny inbound traffic from the known malicious IP ranges.
  • Configure the RDS security group to allow inbound on port 3306 from the entire VPC CIDR. Add a network ACL on the public subnet to deny inbound traffic from the known malicious IP ranges.
  • Configure the RDS security group to allow inbound on port 3306 from 0.0.0.0/0. Add a network ACL on the database subnet to deny inbound traffic from the known malicious IP ranges.
Show Explanation

- Why correct (B): The RDS security group referencing the application tier's security group ensures only instances attached to that specific security group can connect on port 3306, enforcing least privilege at the database layer. The network ACL deny rule on the public subnet blocks known malicious IP ranges before traffic reaches the ALB, since NACLs are stateless and evaluated before security groups. This combination provides defense in depth at both the subnet and instance level. - Why A is wrong: While the RDS security group rule is correct, placing the NACL deny rule on the database subnet instead of the public subnet does not block malicious IPs from reaching the ALB. Malicious traffic targets the public-facing ALB in the public subnet, so the NACL must be on the public subnet. - Why C is wrong: Allowing the entire VPC CIDR on port 3306 violates least privilege. Any instance in the VPC could reach the database, not just the application tier. Security group references are more restrictive and appropriate. - Why D is wrong: Allowing 0.0.0.0/0 on port 3306 opens the database to all traffic, which is a severe security misconfiguration. Even though the database is in a private subnet, this violates the principle of least privilege and defense in depth.

  • Deploy the application in a single AZ with daily EBS snapshots replicated to another Region. Use Route 53 simple routing to point to the primary Region. In the event of a failure, manually launch instances from snapshots in the DR Region.

- Why correct (C): Aurora Global Database provides managed cross-Region replication with typical lag under 1 second, enabling RPOs of seconds. A company runs a three-tier e-commerce application on Amazon ECS with an Aurora PostgreSQL database. A new release includes both application changes and a database schema migration that adds columns used by the new code.   • Use an ECS rolling update with minimum healthy percent set to 100% and deploy the schema migration before the code change
  • Use an all-at-once deployment with a CloudFormation stack update and a rollback trigger on CloudWatch alarms
  • ✓ Use AWS CodeDeploy blue/green deployment for ECS with backward-compatible additive schema changes and CloudWatch alarm-triggered automatic rollback
  • Use a canary deployment shifting 10% of traffic first, and apply the schema migration only after the canary is validated

- Why A is wrong: A rolling update replaces tasks in batches but does not maintain two full environments simultaneously. If the schema migration is applied, canary does not isolate the old code from the new schema any better than blue/green, and CodeDeploy's blue/green mode provides faster full rollback than progressively draining canary traffic.

Quiz 15 — Sample Question
Domain 3: Continuous Improvement for Existing Solutions
Select TWO. A financial institution operates 50 RDS databases across multiple accounts. Application teams currently store database credentials in configuration files on EC2 instances. A recent security audit flagged that credentials are rarely rotated and some are shared across environments. The CISO requires automated credential rotation and reduced exposure risk. Which TWO actions BEST address these requirements?
  • Store credentials in EC2 environment variables and rotate them manually during quarterly maintenance windows
  • Extend credential rotation intervals to 180 days to reduce operational disruption from rotation failures
  • ✓ Migrate credentials to AWS Secrets Manager with automatic rotation enabled on a 30-day schedule
  • Hardcode credentials in the application source code and encrypt the repository with AWS KMS
  • ✓ Enable IAM database authentication for supported RDS engines to eliminate password-based access
  • Create a single master credential shared across all databases to simplify rotation management
Show Explanation

- Why correct (C and E): AWS Secrets Manager provides automatic rotation of secrets on a configurable schedule (e.g., every 30, 60, or 90 days) using Lambda rotation functions. For RDS, Secrets Manager offers built-in rotation templates that update both the secret and the database credential automatically without application downtime. IAM database authentication eliminates passwords entirely for supported engines (MySQL, PostgreSQL on RDS/Aurora) by using short-lived IAM tokens, removing the risk of static credential exposure altogether. Together, these two approaches address different parts of the credential lifecycle—rotating what must remain a secret and eliminating secrets where possible. - Why A is wrong: Storing credentials in environment variables does not provide rotation, auditing, or encryption at rest. If the environment is compromised (e.g., container escape, EC2 metadata exposure), credentials are exposed in plaintext. This violates secrets management best practices. - Why B is wrong: Extending rotation intervals to 180 days increases the window of exposure if credentials are compromised. Security best practices recommend shorter rotation intervals, not longer. The goal is to reduce risk from credential theft, which requires more frequent rotation. - Why D is wrong: Hardcoding credentials in application source code is one of the most dangerous practices—credentials end up in version control, CI/CD logs, and container images. Any developer or system with repository access can read them. This is the opposite of secure credential management. - Why F is wrong: A shared master credential across all databases creates a single point of failure. If that one credential is compromised, all 50 databases are exposed simultaneously. Least privilege requires separate credentials per database or service.
